# Распределенные системы

## Что это такое?

Распределенная система - это набор независимых компьютеров (узлов), соединенных сетью, которые работают вместе для достижения общей цели, выглядя для пользователя как единое целое. 

## Требования к системе

1. Функциональные - как должна вести себя система с точки зрения бизнеса, как функции должна бвыполнять система.

2. Нефункциональные - масштабируемость, стабильность, отказоустойчиво и т.д.


## Основные свойства распределенных систем

1. **Производительность**
    - low latency - системы с маленькой задержкой ответа - важен маленький latency при потери допустимой пропускной способности

    - high throughput - системы с высокой пропускной способностью - нужен максимальный throughput по какому-то определенному latency

2. **Масштабируемость** - способность системы расти без потери производительности, а также без необходимости менять программную архитектуру  
    
    - вертикальное масштабирование
    Проблемы: 
        - стоимость растет нелинейно относительно нагрузки
        - невозможно масштабироваться бесконечно. быстро упираемся в потолок произовдительности
        - перенос данных скорее всего будет сопровождаться _downtime_'ом приложения
    
    - горизонтальное масштабирование

    - масштабирвоание по времени* - различные данные имеют различные требования к обновлению - это позволяет отложить часть обработки данных до более удобного случая

    Параметры масштабируемости:
    - rps
    - трафик (количество Bps)
    - число коннекшенов

    Read / Write ration - в системе больше записи или чтения

    Распределение касательно видов нагрузки:
    - Data-intensive - системы, которые оперируют большими объемами данных
    - Compute-intensive - нужно много вычислительных операций (online игры, движок по подбору рекламы)

3. **Отказуустойчивость системы** - способность системы быть доступной
    - избыточность на аппаратном уровне (запасная машина, запасной ДЦ и т.д.)
    - обнаружение и обработка отказа на программном уровне
    - прогнозирование и предотвращение отказов (observability)

4. **Доступность** - когда пользователи могут взаимодействовать с системой, получая корректные ответы за приемлемое время (number of nines)

5. **Надежность** - способность работать без сбоев и ошибок в течении длительного времени

6. **Удобность поддержки**

7. **Безопасность**


## Шаги по проектированию системы

1. Уточнить функциональные и нефункциональные требования
2. Задизайнить высокоуровнево систему, основываясь только на функциональных требованиях. Обязательно знать фидбэк от stackholder'ов.
3. Реализовать нефункциональные требования. 
4. Дополнительно: безопасность, аутентификация, количество денег на инфру, количество серверов и тд
